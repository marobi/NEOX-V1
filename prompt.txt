We are continuing the NEO6502_MMU / NEOX OS project.

Context:
- Target CPU: W65C02.
- Toolchain: ca65/ld65.
- Kernel language: 6502 assembly.
- RP2350 is the I/O co-processor.
- Syscalls are direct jump-table calls, not BRK.
- Kernel was moved from $E000 to $8000 because the old 4 KB kernel space was too small.
- Current layout direction:
  $8000-$BFFF  kernel
  $C000-$CFFF  syscall veneer
  $D000-$DFFF  RP I/O
  $E000-$EFFF  MICMON / monitor
  $F000-$FFFF  BIOS

Coding preferences:
- Generate concrete ca65-compatible 65C02 assembly.
- Do not use pseudocode unless explicitly asked.
- Use K&R-like compact style.
- Do not use LF in strings. Strings must end with CR only:
  .byte "text", 13
- Avoid long relative branches where likely to exceed 6502 branch range; use jmp when safer.
- Do not casually change Makefile/build structure unless asked.
- Keep temporary debug code clearly marked.

Important current OS status:
- Scheduler, process table, console I/O, FD layer, dup/dup2/close/read/write all work.
- PID 0 is special idle/supervisor/monitor fallback.
- PID 0 must not be treated as a normal schedulable user process.
- Console read can block/yield.
- Monitor/MICMON integration works.
- Process states include RDY/RUN/BLK etc.
- Wait reasons/objects are already displayed in ps.
- Existing debug fields in scheduler should remain for now because pipes will introduce blocking/yielding.

Synchronization model:
- New lock mechanism exists in include/lock.inc using W65C02 TSB/TRB macros.
- Locks are shared memory bytes, not zero page.
- LOCK_ACQUIRE / LOCK_RELEASE clobber A and flags.
- LOCK_RELEASE also sets C before returning; callers must set desired return flags after release.
- Rules:
  - Do not hold fd_lock across device backend calls.
  - Do not hold pipe_lock across sched_yield/blocking wait.
  - Do not hold rp_lock across indefinite yield unless deliberately owning an async transaction.
  - Do not convert sched_lock into LOCK_ACQUIRE. sched_lock remains scheduler policy gate.
  - Scheduler IRQ must not spin on fd_lock/pipe_lock/rp_lock.

Important issue for pipes:
- fd_close_current_locked currently exists for dup2 and only updates FD tables/refcounts. It does not perform backend close/wakeup.
- Before or during pipe implementation, close should probably be split into:
  fd_close_pid_core:
    caller holds fd_lock
    detaches fd
    decrements refcount
    returns object/type and whether last close or endpoint wakeup is needed
  fd_close_pid:
    acquires fd_lock
    calls core
    releases fd_lock
    performs backend/pipe close effects outside fd_lock or with pipe_lock as needed
  fd_dup2:
    can call the locked core without recursively acquiring fd_lock

Initial pipe semantics:
- pipe() creates two FDs:
  read end: FD_FLAG_READ
  write end: FD_FLAG_WRITE
- Both FDs reference a shared open pipe object or paired endpoint objects, whichever is simpler and correct.
- read on empty pipe:
  if writer exists: block current process on pipe read wait
  if no writer exists: return EOF, length 0
- write when no reader exists:
  return EPIPE if defined, otherwise EIO initially
- write when buffer full:
  first implementation may return short write or EAGAIN if defined
  later implementation should block writer
- close read end:
  mark reader closed
  wake blocked writers
- close write end:
  mark writer closed
  wake blocked readers
- Never hold pipe_lock across sched_yield.
- Never hold fd_lock while blocking.
- Avoid holding fd_lock and pipe_lock together unless order is defined. Preferred order if needed: fd_lock first for FD/open-object lookup, then release fd_lock before taking pipe_lock for pipe data movement.

Need decide pipe object design:
Option A:
- One open object of type OBJ_PIPE represents a pipe endpoint with flags saying read/write end.
- But read and write FDs need endpoint distinction.

Option B preferred:
- Two open objects:
  one OBJ_PIPE read endpoint
  one OBJ_PIPE write endpoint
- Both point to same pipe index.
- Need new open_pipe[OPEN_MAX] and open_pipe_mode[OPEN_MAX] or reuse open_flags.
- read FD references read endpoint object.
- write FD references write endpoint object.
- Pipe table tracks reader/writer open counts or endpoint state.
